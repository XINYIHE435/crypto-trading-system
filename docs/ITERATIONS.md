## v0.1 - MVP验证阶段

### 📅 时间线
**2026年5月10日 - 5月31日**（3周）

### 🎯 迭代目标
验证跨交易所套利策略的可行性，构建最小可用产品（MVP）。

### 📦 功能清单
- ✅ 数据采集模块
  - 支持Binance、KuCoin、Bybit三个交易所
  - 自动下载K线数据（30分钟周期）
  - 自动下载资金费率数据
  - 数据对齐和清洗
  
- ✅ 策略引擎
  - 实现跨交易所套利策略
  - 3种套利类型：价差套利、资金费率套利、组合套利
  - 6种开仓条件，12种平仓条件
  
- ✅ 回测系统
  - 支持历史数据回测
  - 计算盈亏、胜率等指标
  - 生成交易记录
  
- ✅ 命令行工具
  - `python main.py download` - 下载数据
  - `python main.py backtest` - 运行回测
  - 支持参数配置（X, Y阈值）

### 📊 测试结果

#### 初始参数配置
```python
X = 0.5%    # 差价触发阈值
Y = 0.1%    # 费率差触发阈值
P = 0.3%    # 盈利目标
Q = 0.5%    # 止损阈值
transaction_fee = 0.05%  # 手续费
```

#### 回测结果
- **交易次数**: 0笔
- **原因**: 参数阈值设置过高

**问题发现**：
- 实际市场价差：0.003% - 0.01%
- 设置阈值：0.5%
- **阈值是实际价差的50-100倍！**

### 💡 关键洞察

#### 洞察1: 市场效率很高
主流币种（BTC/ETH）在正规交易所之间的价差极小（<0.02%），说明市场非常有效。

#### 洞察2: 参数盲调不可行
不了解实际市场数据就设置参数，导致策略完全无法触发。

#### 洞察3: 需要调试工具
缺少调试信息，无法了解为什么没有交易，需要添加详细日志。

### 🔄 决策与行动
1. **立即行动**: 降低参数阈值，使用X=0.01%, Y=0.01%
2. **技术债务**: 添加调试模式和详细日志
3. **下一步**: 深入分析为什么策略亏损

#### Step 4: 根因分析（6月6日）

##### 问题1: 手续费完全吞噬利润 ⚠️⚠️⚠️

**手续费计算**：
```
每笔套利涉及4次交易：
1. 开仓A交易所: $80,000 × 0.0123 × 0.05% = $0.50
2. 开仓B交易所: $80,000 × 0.0123 × 0.05% = $0.50
3. 平仓A交易所: $80,000 × 0.0123 × 0.05% = $0.50
4. 平仓B交易所: $80,000 × 0.0123 × 0.05% = $0.50
总计: $2.00
```

**价差收益计算**：
```
开仓价差: 0.015%
平仓价差: 0.005%
价差收敛: 0.01%
收益 = 0.01% × $1,000 = $0.10
```

**结论**: $0.10 << $2.00，数学上不可行！

##### 问题2: 持仓规模太小

```
当前: $1,000 (10%仓位)
需要: 至少$20,000才能覆盖手续费
```

##### 问题3: 持仓时间过短
- 大部分交易仅0.5小时（30分钟）
- 价差还没充分收敛就被止盈/止损

##### 问题4: 资金费率收益异常
- 交易#4显示资金费率收益为-$0.07（负数）
- 逻辑可能有误

### 📊 数据洞察

#### 洞察1: 市场价差本质上太小
从1424根K线数据（30天）分析：
- 价差中位数: 0.005%
- 价差均值: 0.007%
- 价差90分位: 0.015%
- 价差最大: 0.025%

**结论**: 99%的时间价差<0.02%

#### 洞察2: 数学上不可行
```
需要价差收敛 = 手续费 / 持仓规模
              = $0.80 / $1,000
              = 0.08%

实际价差范围 = 0.005% - 0.02%
```

**0.08% >> 0.02%**，即使价差完全收敛也无法覆盖成本！

#### 洞察3: 策略不适合主流币
BTC/ETH等主流币市场效率极高，价差太小，不适合跨交易所套利。

**适合的场景**：
- 小市值币种（价差0.1%-1%）
- 期现套利（价差0.05%-0.5%）
- 事件驱动（上新、维护）

   - 便于回顾和总结

### ⚠️ 需要改进的地方

1. **市场调研不足**
   - 没有提前验证市场价差范围
   - 导致策略方向错误
   - **教训**: 先调研，再开发

2. **参数设置盲目**
   - 初始参数完全脱离实际
   - **教训**: 基于真实数据设置参数

3. **成本估算不准**

   - 只开发了一种策略

   - **教训**: 多策略并行验证

### 💡 关键教训

#### 教训1: 先验证，再开发
不要花几周时间开发一个未经验证的策略。应该：
1. 先做市场调研（1天）
2. 快速原型验证（2天）
3. 确认可行性后再深入开发

#### 教训2: 数学先行
任何量化策略都要先算账：
- 预期收益是多少？
- 成本是多少？
- 能否覆盖成本？
- 风险收益比如何？

#### 教训3: 工具投资值得
虽然策略失败了，但开发的工具链很有价值：
- 诊断系统
- 可视化工具

#### 教训4: 拥抱失败
v0.1和v0.2虽然策略失败，但：
- 快速识别了问题
- 学到了宝贵经验

### 🔍 诊断过程

#### Step 1: 系统性诊断（6月10-13日）

**开发内容**：
创建了7步诊断工具 `diagnose_arbitrage.py`，系统性排查策略问题：

1. ✅ **验证套利方向**
   - 检查开仓方向判断逻辑
   - 验证盈亏计算公式
   - 随机抽样5笔交易人工验证
   - **结论**: 代码逻辑100%正确

2. ✅ **检查手续费**
   - 重新计算毛收益和净收益
   - 验证手续费是否重复扣除
   - 统计手续费占比
   - **结论**: 手续费计算正确，但是致命问题

3. ✅ **检查资金费率方向**
   - 验证做空A时是否收取正费率
   - 检查费率方向与持仓方向的匹配
   - **结论**: 基本正确，影响极小（$0.01/笔）

4. ❌ **验证市场机会** 【关键发现】
   - 统计价差分布（均值、中位数、分位数）
   - 绘制价差直方图
   - 对比价差与手续费
   - **结论**: 市场根本不存在套利机会！

5. ✅ **检查净敞口**
   - 计算持仓的美元价值差异
   - **结论**: 存在0.12%的净敞口，影响可忽略

6. ❌ **扩展到90天数据**
   - 尝试下载90天数据验证
   - **结论**: Binance API返回451错误（地区限制）

7. ✅ **生成诊断报告**
   - 导出前20笔交易明细
   - 生成完整诊断报告

#### Step 2: 参数扫描优化（6月13日）

**测试配置**：
```python
# 激进策略
X = 0.005%, Y = 0.005%
结果: 278笔, -$201, -2.01% ❌

# 基线策略  
X = 0.01%, Y = 0.01%
结果: 164笔, -$117, -1.17% ❌

# 中等策略
X = 0.02%, Y = 0.015%
结果: 20笔, -$14, -0.14% ❌

# 保守策略
X = 0.1%, Y = 0.05%
结果: 0笔, $0, 0.00% ⚪
```

**发现**：
- 激进参数 → 更多交易 → 更多手续费 → 更大亏损
- 保守参数 → 无交易（市场价差不够大）
- **无论如何调整参数都无法盈利**

#### Step 3: 数学证明（6月13日）

**盈亏平衡分析**：
```
要实现盈亏平衡，需要:
  价差收敛 ≥ 手续费成本
  价差收敛 ≥ 0.08%

市场现实:
  价差均值: 0.0063%
  价差95分位: 0.0162%
  价差最大值: 0.0367%

结论:
  0.0367% < 0.08%
  
即使在最极端情况（价差最大值完全收敛到0），
也无法覆盖手续费成本。

这是数学上不可能盈利的！
```
